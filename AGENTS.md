本次为大型代码，不使用副本！

提供“开箱即用”且“高度可迁移”的开发体验。开发者习惯直接在 Windows 系统的 IDE 中点击运行，项目会在多台不同的 Windows 电脑上运行。Agent 必须自动处理环境配置、动态路径映射和 Windows 兼容性，确保代码生成后无需修改即可跨设备直接运行。

我希望代码质量非常高，风格简洁、优雅、专业。

请额外遵守以下要求：

1. 简洁优先，但不要为了少写几行而牺牲可读性
   不要写晦涩的一行流、复杂嵌套表达式、过度链式调用。代码应该让人一眼能看懂意图。
2. 设计要克制
   不要过度封装，不要为了“看起来高级”而引入过多类、继承、装饰器、元编程或复杂设计模式。
   只有当拆分真的能提升可维护性时，才进行抽象。
3. 命名要精确
   函数名、变量名、类名要直接表达用途。宁可名字稍长，也不要含糊。
4. 控制函数长度
   如果一个函数开始同时处理界面、数据、文件、模型推理、格式转换等多种职责，请主动拆分。
5. 降低复杂度
   优先减少嵌套层级。可以使用提前返回、辅助函数、清晰的数据结构，让主流程像说明书一样清楚。
6. 统一代码风格
   同类逻辑使用一致的写法。不要同一个程序里一会儿用一种风格，一会儿用另一种风格。
7. 异常处理要自然
   对用户可能遇到的问题给出清楚提示，例如模型路径不存在、图片未加载、预测为空、CUDA 不可用、依赖库未安装。
   但不要用过大的 try/except 把真正的 bug 吃掉。
8. 保持主流程干净
   复杂细节放到命名清楚的辅助函数里，让核心流程读起来接近这样：
   - 获取当前图片
   - 读取模型配置
   - 执行预测
   - 转换为标注
   - 弹窗供用户选择
   - 插入选中的标注
   - 刷新界面
9. 优雅不是炫技
   请避免“聪明但难维护”的写法。我的偏好是：清楚、稳定、简洁、长期好改。
10. 修改完成后，请主动做一次代码质量复盘
    请指出：
    1. 哪些地方为了简洁做了拆分
    2. 哪些地方避免了过度设计
    3. 哪些函数是后续最容易扩展的入口
    4. 是否还有值得进一步重构的部分

代码质量强制要求：

1. 所有函数、方法、类都添加完整的类型注解，禁止无意义使用Any类型
2. 每个公开函数和类都编写谷歌风格的docstring，说明功能、参数、返回值和异常
3. 所有硬编码的固定值提取为大写常量，统一放在文件头部管理；实验性质的应该更靠前
4. 对文件读写、路径解析添加完整异常处理，错误信息清晰可定位
5. 避免深层嵌套，优先使用提前返回简化分支逻辑
