检查时间：2026-07-07

## 1. 项目总体结论

本项目是一个智慧工地数字孪生监控平台，整体架构为：

- 前端：Three.js + ECharts + ES6 Modules
- 后端：FastAPI + WebSocket + OpenCV
- AI 检测：Ultralytics YOLOv8 + YOLOv8-Pose
- 几何判断：Shapely 多边形区域检测
- 坐标映射：OpenCV 单应性矩阵

后端主要负责视频源接入、模型加载、目标检测、跌倒判断、危险区域入侵判断、像素坐标到 3D 场景坐标映射，以及通过 WebSocket 向前端推送检测事件、人员轨迹和视频帧。

## 2. 后端结构梳理

后端入口文件为：

```text
backend/app.py
```

该文件只负责创建 FastAPI 应用、注册 HTTP 路由、注册 WebSocket 端点，并在生命周期启动时初始化检测组件。

### 2.1 核心后端模块

| 文件                                       | 作用                                 |
| ---------------------------------------- | ---------------------------------- |
| `backend/app.py`                         | FastAPI 后端入口，注册路由和生命周期             |
| `backend/ws.py`                          | WebSocket 推流端点，推送事件、轨迹和视频帧         |
| `backend/services/model_loader.py`       | 读取模型配置、解析模型路径、加载检测器、维护运行时状态        |
| `backend/services/detection_pipeline.py` | 执行真实视频帧检测，合并安全帽、烟火、跌倒、入侵检测结果       |
| `backend/services/frame_codec.py`        | 将视频帧编码为 JPEG/base64，供 WebSocket 推送 |
| `backend/services/mock_stream.py`        | 模型或视频不可用时生成模拟数据和模拟视频帧              |
| `backend/detector.py`                    | YOLOv8 目标检测封装，支持安全帽和烟火模型           |
| `backend/pose_analyzer.py`               | YOLOv8-Pose 姿态估计和跌倒规则判断            |
| `backend/video_source.py`                | 管理文件、RTSP、USB 摄像头等视频源              |
| `backend/coord_mapper.py`                | 通过单应性矩阵将像素坐标映射到 3D 场景坐标            |
| `backend/zone_monitor.py`                | 基于 Shapely 判断人员是否进入危险区域            |

### 2.2 HTTP API

| API                  | 作用                         |
| -------------------- | -------------------------- |
| `/api/health`        | 查看后端状态、模型加载状态、是否进入 mock 模式 |

| `/api/videos`        | 查看当前已接入的视频源                |

### 8.2 跌倒数据转换脚本存在硬编码路径

```text
C:\Users\23184\Desktop\行人跌倒检测
```

以下脚本中存在 API key 字符串：

- `backend/download_roboflow.py`
- `backend/download_multi.py`

建议后续改为从环境变量读取，避免泄露和失效。

### 8.4 跌倒模型路线需要统一说明

项目中存在两条跌倒检测路线：

- 已训练的目标检测模型：`fall_detect.pt`
- 当前实际使用的方案：`yolov8n-pose.pt` + 姿态几何规则

当前后端配置采用第二种方案。报告、答辩或说明文档中应明确写为“最终采用姿态估计方案判断跌倒”，避免误以为线上直接加载 `fall_detect.pt`。

### 8.5 后端代码质量仍可继续提升

当前后端可以运行，但如果严格按照项目 `AGENTS.md` 的质量要求，还需要补充：

## 10. 复盘

### 10.1 哪些地方为了简洁做了拆分

从现有结构看，后端已经将入口、模型加载、检测流水线、视频源、区域检测、坐标映射和 WebSocket 推流拆成了独立模块。这种拆分使 `app.py` 保持简洁，核心业务逻辑集中在命名清楚的服务模块中。

### 10.2 哪些地方避免了过度设计

后端没有引入复杂的类继承或多层抽象。大多数模块只围绕一个明确职责组织，例如 `VideoSourceManager` 只管理视频源，`ZoneMonitor` 只处理区域判断，`CoordMapper` 只负责坐标映射，整体比较克制。

### 10.3 哪些函数是后续最容易扩展的入口

后续最值得作为扩展入口的函数包括：

- `init_components()`：新增模型加载逻辑。
- `run_all_detectors()`：接入更多目标检测模型。
- `process_real_frame()`：新增事件类型、轨迹字段或告警策略。
- `FallDetector.check_fall()`：优化跌倒判定规则。
- `VideoSourceManager.add_source()`：扩展视频源接入方式。

### 10.4 是否还有值得进一步重构的部分

值得进一步重构的部分主要集中在训练和数据准备脚本：

- 将硬编码路径改为命令行参数。
- 将数据集路径统一为相对路径或当前项目路径。
- 补充独立 `train.py`，把训练命令、模型名称、数据集配置和输出目录纳入统一管理。
- 将 Roboflow API key 改为环境变量。
- 对后端旧模块补齐类型注解和 Google 风格 docstring。
